## Squash Strategy

### When to Squash

Consider squashing when either threshold is hit:

| Threshold | Recommendation |
|---|---|
| Migration count | > 50 files total |
| Age of oldest | > 6 months since last edit to the oldest batch |
| Reset time | `supabase db reset` takes > 30 seconds |

Given the current state (38 files, oldest from Feb 2024), squashing is **advisable but not urgent**. The oldest 17 migrations (Feb 2024) are stable and never modified -- they are ideal squash candidates.

### Steps to Squash

1. **Identify the cutoff**: Choose a date or migration name. All migrations up to (and including) the cutoff will be squashed.
   - Example: squash everything before `202602180001_admin_infra.sql` (Feb 2026 boundary)
   - The cutoff should always be the last migration in a stable, released batch.

2. **Dry-run the consolidation**:
   ```bash
   supabase db dump --file consolidated_base.sql
   ```
   This produces the full current schema as a single file.

3. **Create a new baseline migration**:
   ```bash
   supabase migration new squash_YYYYMMDD_baseline
   ```
   Copy the content of `consolidated_base.sql` into the new migration file, but add a prefix comment:
   ```sql
   -- Squashed migration: replaces 20240213_* through 20260218_* (XX files)
   -- Created: YYYY-MM-DD
   -- Do not edit this file directly. Edit source migrations and re-squash.
   ```

4. **Remove squashed migrations**: Delete the individual files from `supabase/migrations/` that are now represented by the baseline.

5. **Update `supabase/migrations/.snapshots/`** if tracked.

6. **Reset and verify**:
   ```bash
   supabase db reset
   supabase db dump --local --file verify_after_squash.sql
   diff consolidated_base.sql verify_after_squash.sql  # should match
   ```

### Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Squash drops a DDL statement | Low | High | Run `diff` between pre- and post-squash dumps |
| Migration history lost for debugging | Medium | Low | Commit message references squashed file list; git preserves history |
| Branch migrations conflict | Low | Medium | Squash on a dedicated branch; rebase feature branches after merge |
| Reset takes longer | Low | Low | Baseline is a single file, so replay is actually faster |

### Rollback Plan

If the squash introduces issues:

1. `git revert` the commit that removed the old migration files
2. Delete the new baseline migration file
3. Run `supabase db reset` to restore from the old individual files
4. Investigate the discrepancy by comparing pre-squash and post-squash `pg_dump` output

A squash is purely a local/file reorganization -- it does not run any SQL against production. Rollback is safe.

## Naming Convention

Current pattern: `YYYYMMDDHHMMSS_description.sql`

**Status**: CONFIRMED -- this is the Supabase-recommended convention. It provides:

- Chronological ordering (lexical sort = date sort)
- Human-readable description
- Uniqueness (timestamp granularity + description)
- Supabase tool compatibility (`supabase migration new` auto-generates this format)

**Rule**: Always use `supabase migration new <name>` to generate the timestamp prefix. Never hand-craft timestamps.

## Migration Workflow

### Local Development

```bash
# Create a new migration
supabase migration new add_donor_wall_field

# Write SQL in the generated file
# File: supabase/migrations/20260628120000_add_donor_wall_field.sql

# Apply to local DB
supabase db reset          # replays ALL migrations from scratch
# OR
supabase migration up      # applies only pending migrations

# Inspect current state
supabase db dump --local --file current_schema.sql
```

### Testing

```bash
supabase db reset          # tears down and rebuilds from migrations
supabase db test           # runs pgTaste tests if configured
```

### CI/CD

Migrations are **manual, not automatic** in CI/CD:

1. Developer creates migration PR
2. Review checklist is completed (see below)
3. PR is merged to main
4. A designated person (lead or infra) applies to production:
   ```bash
   supabase db push --linked
   ```
   This runs only pending migrations against the linked Supabase project.

**Never** configure automatic migration application in CI. A mistaken push can lock tables or drop data.

## Schema Drift Detection

Drift occurs when the local database state differs from what the migration files produce. Common causes:

- Hand-editing the database via `psql` or Supabase Studio without creating a migration
- Running `supabase db dump` and treating the output as authoritative
- Two developers making conflicting changes

### Detection Methods

1. **`supabase db diff --linked`**: Compares local migrations against the linked Supabase project. Shows any DDL that exists remotely but not in migrations.

2. **`supabase db dump --local > local.sql` vs migration replay**: Reset local DB, dump schema, then compare dump to the migration files logically.

3. **`supabase migration list`**: Shows which migrations have been applied. If migrations are missing or out of order, this flags it.

### Fixing Drift

- Minor drift: Create a new migration that brings the schema in line with migrations.
- Major drift: Rebase the branch or reset the branch database.

## Data Migrations

Data migrations move or transform existing rows. They are riskier than schema migrations.

### Guidelines

| Rule | Explanation |
|---|---|
| **Always additive** | INSERT new rows or ADD COLUMN. Never DELETE or DROP COLUMN in the same migration that creates a replacement. Separate into two migrations. |
| **Backfill in batches** | For large tables, use `UPDATE ... LIMIT 1000` loops with `pg_sleep`. Never lock a whole table in one transaction. |
| **Idempotent** | Use `IF NOT EXISTS` / `OR REPLACE` so re-running the migration is safe. |
| **Test on copy** | Restore a production backup to a branch and run the data migration there first. |
| **Timestamp columns** | Always set `created_at` / `updated_at` explicitly in data migrations so they are auditable. |

### Template for a Data Migration

```sql
-- Migration: backfill_donor_wall_flags
-- Idempotent: yes

-- Step 1: add column (safe, no lock on reads)
ALTER TABLE IF EXISTS public.donations
  ADD COLUMN IF NOT EXISTS show_on_wall BOOLEAN DEFAULT true;

-- Step 2: backfill -- only touches rows where the column is STILL NULL
-- (handles interruption mid-way)
UPDATE public.donations
SET show_on_wall = true
WHERE show_on_wall IS NULL
  AND organisation_id IS NOT NULL;
```

## Review Checklist

Before merging a pull request that contains a new migration, verify each item:

- [ ] **Idempotent**: Uses `IF NOT EXISTS`, `OR REPLACE`, `ADD COLUMN IF NOT EXISTS` where applicable
- [ ] **No destructive operations**: No `DROP COLUMN`, `DROP TABLE`, or `DELETE` without explicit approval
- [ ] **Proper naming**: Timestamp generated by `supabase migration new`, not hand-crafted
- [ ] **Single concern**: One migration file handles one logical change
- [ ] **Drift check**: `supabase db diff --linked` shows no unexpected drift
- [ ] **Reset test**: `supabase db reset` completes successfully on a clean local DB
- [ ] **Rollback known**: The author has stated how to revert this migration (reverse SQL or next migration)
- [ ] **Data migrations flagged**: If the migration includes DML, it has been approved by the lead

## Emergency Rollback

### Scenario: A bad migration was applied to production

```bash
# 1. Identify the problematic migration
supabase migration list

# 2. Check if any data depends on the new schema
#    If the migration only ADDED columns/tables, rollback is just:
supabase db push --linked --version <previous-migration-timestamp>

# 3. If the migration dropped or altered columns, data recovery is needed:
#    - Restore from backup if available
#    - Or write a compensating migration that recreates the lost structure

# 4. Audit what happened
supabase db dump --linked > post_incident_schema.sql
```

### Key Principles

- **Schema-only rollback**: `supabase db push --linked --version <target>` replays from scratch up to the target version. This is safe for schema-only changes.
- **Data-loss rollback**: Never try to reverse a destructive migration automatically. Restore from backup or write a forward-fix migration.
- **Communication**: Announce the rollback on the team channel. Tag the PR author. File a postmortem.

### Preventing Bad Migrations

- Always test on a branch database first
- Run `supabase db reset` locally before merging
- Review checklist (above) must be complete
- Consider wrapping high-risk migrations in a transaction and testing manually
